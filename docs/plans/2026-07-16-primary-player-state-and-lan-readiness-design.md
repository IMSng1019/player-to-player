# 主客户端玩家状态与 LAN 就绪设计

## 背景

实测中，玩家正常进入物理服务器后被指派为主客户端，随后按设计断开物理服并启动本地集成服务器。切换完成后画面接近全黑，玩家坐标从物理服上的正常位置变成 `(0, 0, 0)` 附近；日志同时出现 `Not a string` 和客户端空玩家对象 NPE。

日志与存档解析确认了两个独立根因：

- 环境中的 `playerdata/<UUID>.dat` 保存了正确位置，但同步下来的 `level.dat` 包含一个存在但内容为空的 `Data.Player`。原版集成服务器会优先读取单人世界所有者的 `Data.Player`，因此绕过正确的 UUID 玩家文件，以默认坐标创建玩家，并在停服时把错误状态再次写回本地存档。
- `GroupHost` 在集成服务器触发 `SERVER_STARTED` 后立即调用 `IntegratedServer.publishServer`。此时本地客户端玩家尚未完成登录，`Minecraft.player` 为 null。原版已经打开 TCP 监听并标记世界已发布，随后在更新玩家权限时抛出 NPE，导致界面显示局域网，但模组没有记录 LAN 端口，也没有发送 `GROUP_WORLD_READY`。

## 已确认决策

- 玩家被指派为主客户端时，以物理服务器的最新玩家 NBT 为权威来源。
- 本地 `primary/data` 只是在组世界运行期间使用的副本，不得覆盖物理服务器权威状态。
- 无法从物理服务器取得或校验玩家 NBT 时，取消本次主客户端切换：玩家保留在物理服，不申请角色、不创建组、不启动本地世界，并收到可重试提示。
- LAN 发布必须等待本地玩家和本地连接真实就绪，不使用固定延时。

## 目标

- 主客户端切换前后维度、坐标、背包、生命值和其他玩家 NBT 状态连续。
- 已经被错误 `(0, 0, 0)` 状态污染的本地存档能够在下一次加入时自动修复。
- 无效或缺失的权威玩家 NBT 不得写入本地存档。
- 获取权威状态失败时不得产生只有服务端登记、没有本地组世界的空组。
- LAN 端口只在本地玩家登录完成后发布，并可靠记录端口和通告组世界就绪。
- 不新增协议消息类型，不提升 `Protocol.VERSION`。

## 非目标

- 本次不重新设计副客户端加入已有主客户端时的跨组玩家数据迁移。
- 本次不改变区块申请、区块上行、环境快照或合并/分离协议。
- 本次不隐藏原版的“局域网”状态。主客户端集成服发布 LAN 是组内隧道接入所必需的正常行为。

## 方案选择

采用“切换前获取权威快照 + 本地存档去除单人玩家内嵌状态 + 条件式 LAN 发布”。

相比仅删除 `Data.Player`，该方案能修复已经污染的本地 `playerdata`；相比 Mixin 修改原版 `PlayerList.load`，该方案不侵入玩家登录核心方法，版本兼容面更小。通过复用现有 `PLAYER_DATA_REQUEST` / `PLAYER_DATA` 消息，协议结构保持兼容。

## 玩家状态数据流

客户端完成环境同步后，按以下顺序继续：

1. 客户端在仍连接物理世界时发送本人的 `PLAYER_DATA_REQUEST`。
2. 物理服务器在服务端主线程查询在线 `ServerPlayer`。玩家在线时调用 `saveWithoutId` 生成实时快照；只有玩家不在线时才回退读取 `playerdata/<UUID>.dat`。
3. 服务端使用原版 gzip NBT 格式返回 `PLAYER_DATA`。
4. 客户端完整解析 NBT，并至少校验：
   - 根标签是复合标签；
   - `Dimension` 是非空字符串；
   - `Pos` 是三个有限数值；
   - 压缩后大小不超过既有限制。
5. 客户端先构建或检查本地主缓存，再把已验证玩家 NBT 原子写入 `playerdata/<UUID>.dat`。
6. 从本地 `level.dat` 和 `level.dat_old` 的 `Data` 复合标签中删除 `Player`，避免原版集成服优先读取陈旧内嵌状态。
7. 只有权威快照和本地文件全部准备成功后，客户端才发送 `ROLE_REQUEST`。
8. 若角色为副客户端，保留已准备的主缓存供之后晋升复用，继续现有 P2P 隧道加入流程。
9. 若角色为主客户端，预置 `GroupRuntime` 并切换到已经准备完成的本地世界。

权威快照和本地文件准备都在 `ROLE_REQUEST` 前完成，可以避免角色已经登记为主客户端后才发现玩家状态或磁盘不可用。额外请求的数据通常只有数 KB；副客户端提前准备一次小型世界骨架的成本，也远小于增加角色取消协议和服务端回滚状态机，并可缩短之后分离晋升的切换时间。

## 本地文件更新

玩家文件和 `level.dat` 修改都采用“同目录临时文件 + 原子替换”：

- 玩家 NBT 先解析为 `CompoundTag`，再由 `NbtIo.writeCompressed` 重新编码，避免直接落盘未验证的网络字节。
- `level.dat` 只删除 `Data.Player`，保留种子、时间、天气、游戏规则和数据包配置。
- `level.dat_old` 同步清理，防止崩溃恢复时重新引入旧的内嵌玩家状态。
- 文件系统不支持原子移动时，回退为同目录覆盖移动；任何一步失败都不启动本地世界。

本地存档可能已经保存了错误玩家文件。本次每次主客户端启动都会用物理服权威快照覆盖本地 UUID 玩家文件，因此无需用户手工删除 `primary/data`。

## 失败处理

以下任一情况都取消本次角色申请和世界切换：

- 请求超时、控制连接关闭或服务端返回错误；
- 服务端既没有在线玩家，也没有磁盘玩家文件；
- NBT 解压、解析或字段校验失败；
- 本地存档构建、玩家文件写入或 `level.dat` 清理失败。

失败时客户端：

- 保持物理服游戏连接和控制连接；
- 保持 `ClientRole.UNASSIGNED` 和空 `groupId`；
- 在聊天栏显示“无法取得或准备物理服务器玩家状态，已取消主客户端切换，请稍后重新进入服务器”；
- 写入包含阶段和异常原因的日志，但不打印玩家 NBT 内容。

服务端未收到 `ROLE_REQUEST`，因此不会创建组或占用区块，不需要新增回滚消息。

## LAN 发布时序

`GroupHost.start` 仍在集成服务器 `SERVER_STARTED` 阶段挂接 P2P 入站会话监听器，但不立即调用 `publishServer`。它只记录待发布的 `IntegratedServer`。

客户端生命周期注册一个轻量的每 tick 就绪检查，仅当以下条件同时满足时执行一次发布：

- 待发布服务器仍是 `Minecraft.getSingleplayerServer()`；
- `Minecraft.player` 非 null；
- 客户端游戏连接非 null；
- `GroupRuntime` 仍接管该集成服务器。

满足条件后调用 `publishServer`，成功时原子记录 `lanPort`，再发送 `GROUP_WORLD_READY`。会话拆除、集成服务器停止或本地世界打开失败时清除待发布服务器和端口，避免旧状态污染下一次世界会话。

该流程等待的是实际状态条件，不依赖机器性能、固定毫秒延时或预计加载时间。

## 线程模型

- 在线玩家 `saveWithoutId` 只在物理服务器主线程执行。
- 磁盘读取、NBT 压缩/解压和本地文件更新在 `ThreadPools.io()` 执行。
- 客户端角色申请主流程继续在 IO 线程执行，不阻塞渲染线程。
- LAN 就绪检查和 `publishServer` 在客户端主线程执行。
- 待发布服务器、LAN 端口等跨线程状态使用 volatile 或既有线程安全机制发布。

## 兼容性

- 复用现有 `PLAYER_DATA_REQUEST` 和 `PLAYER_DATA`，消息编号与 JSON 字段保持不变。
- `PLAYER_DATA_REQUEST` 的服务端实现从“只读磁盘”增强为“在线快照优先、磁盘兜底”，旧客户端仍可正常消费响应。
- 不修改 `Protocol.VERSION`、Fabric 元数据、Mixin 声明或消息类型。

## 测试策略

自动测试采用真实 gzip NBT 和临时目录，不使用只验证 mock 调用次数的测试：

- 有效权威玩家 NBT 能通过校验并原子写入本地 UUID 玩家文件。
- 空 `Dimension`、错误类型的 `Dimension`、缺失/长度错误/非有限值的 `Pos` 被拒绝。
- 无效权威 NBT 不修改现有本地玩家文件和 `level.dat`。
- `level.dat` 与 `level.dat_old` 中空的、陈旧的或有效的 `Data.Player` 都被删除，其他 `Data` 字段保持不变。
- 已污染的本地玩家文件被权威玩家状态覆盖。
- 本地准备任一步失败时，启动流程不调用角色申请或本地世界切换。

完成后运行：

```powershell
.\gradlew.bat test
.\gradlew.bat build
```

实测场景：

1. 玩家在物理服记录一组明显非零坐标和背包内容。
2. 等待指派为主客户端并切换到本地组世界。
3. 验证切换前后维度、坐标、背包和生命值一致。
4. 验证日志不再出现 `Not a string`、`Minecraft.player` NPE，并出现 LAN 端口发布及 `GROUP_WORLD_READY` 发送记录。
5. 人为让玩家数据请求失败，验证玩家仍留在物理服、未创建组，并收到重试提示。
