# 版本化不可变环境快照设计

## 背景

当前服务端在启动时扫描一次环境目录，生成包含文件 SHA-256 与大小的环境清单；客户端下载文件时，服务端却直接读取实时源文件。`world/level.dat` 等运行期文件会在服务端运行期间变化，因此客户端可能拿到旧清单和新文件内容，最终因大小或哈希不一致而反复同步失败。环境同步失败又会阻断角色申请，玩家会停留在世界 Tick 已挂起的物理服务端。

本设计要求环境清单和下载内容永远来自同一份不可变快照，并在服务端环境发生变化后实时生成和原子发布新快照。

## 目标

- 每份环境清单绑定唯一的 `snapshotId`。
- 客户端下载的每个文件块都来自该 `snapshotId` 对应的不可变内容。
- 环境文件变化后约 1 秒内开始构建新快照，并以 60 秒全量校验兜底文件监听漏报。
- 新快照发布不影响正在使用旧快照下载的客户端。
- 静态大型文件跨快照复用，避免每次 `level.dat` 变化都复制整套服务端环境。
- 服务端和中转服务端使用相同的快照分发机制。
- 快照构建失败时继续提供上一份有效快照，不发布半成品。

## 非目标

- 不在本次修改中改变环境文件的包含与排除语义。
- 不实现跨服务端共享的远程对象存储。
- 不支持协议版本 1 的客户端继续连接新服务端；本次会提升控制协议版本。

## 方案选择

采用版本化内容寻址快照：文件内容以 SHA-256 为键保存为不可变 Blob，快照清单只记录相对路径到 Blob 的映射。相比整目录复制，该方案只为实际变化的内容增加磁盘占用；相比按连接隐式绑定快照，显式 `snapshotId` 能正确支持断线、重试、中转与并发下载。

## 目录结构

服务端和中转端分别在自己的 `player-to-player` 根目录下维护快照仓库：

```text
player-to-player/environment-snapshots/
  blobs/
    <sha256>                 # 不可变文件内容
  manifests/
    <snapshotId>.json        # 完整快照清单
  staging/
    <buildId>/               # 构建中的临时内容，发布前不可见
```

快照仓库本身位于 `player-to-player/` 下，继续命中现有环境扫描排除规则，不会递归进入环境清单。

## 核心组件

### EnvironmentSnapshot

不可变值对象，包含：

- `snapshotId`：完整未过滤清单的全局 SHA-256。
- `EnvironmentManifest manifest`：相对路径到内容哈希和大小的映射。
- `createdAtMillis`：创建时间。
- `lastAccessMillis`：最后一次被清单请求或文件请求访问的时间。

目标端过滤只生成清单视图，不改变底层 `snapshotId`。主客户端、副客户端和中转端看到的过滤哈希可以不同，但都引用同一份完整快照。

### EnvironmentSnapshotManager

负责快照的构建、发布、查询、监听和回收：

- 启动时同步触发首轮后台构建；首份快照发布前 `envReady=false`。
- 递归注册 `WatchService`，只对环境扫描规则包含的路径标记 dirty。
- 收到变化后执行 1 秒防抖，同一时间只允许一个构建任务运行。
- 构建期间再次变化时保留 dirty 标记，当前构建结束后立即补建下一版。
- 每 60 秒执行一次全量元数据校验，发现监听漏报时触发构建。
- 使用单个 volatile/current 引用原子发布完成的快照。
- 按 `snapshotId` 查询保留快照，并在访问时刷新最后访问时间。
- 回收超过保留时间且不是当前版本的清单，再删除没有任何保留清单引用的 Blob。

### EnvironmentSnapshotBuilder

构建流程如下：

1. 使用与 `EnvironmentScanner` 相同的包含和排除规则枚举源文件。
2. 对每个文件记录复制前的大小、修改时间和文件标识。
3. 将源文件复制到 staging 临时文件，同时流式计算 SHA-256 和实际大小。
4. 再次读取源文件元数据；若复制期间发生变化，删除临时文件并重试该文件，最多三次。
5. 将临时文件原子移动到 `blobs/<sha256>`；目标已存在时直接复用并删除临时文件。
6. 全部文件成功后构造完整清单，以清单全局哈希作为 `snapshotId`。
7. 原子写入 manifest 文件，最后由管理器发布新快照。

若任一文件持续变化或读取失败，本轮构建失败，不更新 current 快照。首轮失败保持 `envReady=false`；后续失败继续提供上一份有效快照。

## 实时刷新

采用用户确认的监听方案：

- `WatchService` 递归监听服务端根目录或中转缓存目录。
- 新目录创建后立即注册该目录及其子目录。
- 变化事件经过 1 秒防抖，避免一次原子保存产生的多个事件触发重复构建。
- `OVERFLOW` 事件直接标记全量 dirty。
- 每 60 秒重新计算轻量元数据指纹，发现差异则触发快照构建。
- 发布后记录新的元数据指纹，内容未变化时不产生新快照。

## 协议调整

`Protocol.VERSION` 从 1 提升到 2，防止新旧节点混连。

### ENV_MANIFEST

应答新增：

```json
{
  "snapshotId": "完整快照哈希",
  "globalHash": "完整快照哈希",
  "filteredHash": "目标端过滤清单哈希",
  "manifest": {}
}
```

### ENV_FILE_REQUEST

请求新增必填字段：

```json
{
  "snapshotId": "完整快照哈希",
  "path": "world/level.dat",
  "offset": 0
}
```

服务端通过 `snapshotId + path` 找到清单条目，再从 `blobs/<entry.sha256>` 读取。响应回显 `snapshotId`，客户端同时核对 `snapshotId`、路径和偏移。

找不到快照时返回 `snapshot_not_found`。客户端收到后放弃当前文件循环，重新拉取清单并重新计算 diff；不得回退读取实时源文件。

## 客户端流程

1. 请求目标端清单并保存 `snapshotId`。
2. 扫描本地环境并计算差异。
3. 每个文件块请求都携带同一个 `snapshotId`。
4. 下载完成后继续执行现有大小与 SHA-256 校验。
5. 快照失效时重新开始本轮 `syncTo`。
6. 主客户端和副客户端两套同步可分别使用各自请求时获得的快照；单次 `syncTo` 内必须保持快照固定。

## 中转端流程

中转端从上级服务端同步到 `proxy-environment` 后，不再直接扫描并实时分发该可变目录，而是由本地 `EnvironmentSnapshotManager` 对缓存目录构建和发布不可变快照。上游同步写入文件期间产生的监听事件会被防抖合并；只有完整快照构建成功后，下游客户端才会看到新版本。

## 快照保留与回收

- 新配置 `envSnapshotRetentionMinutes`，默认 120 分钟，最小 10 分钟。
- 当前快照永久保留。
- 非当前快照从最后一次访问起保留配置时长。
- 清单回收后执行 Blob 引用扫描，只删除不再被任何保留清单引用的 Blob。
- staging 中超过保留时长的残留目录在启动和定期回收时删除。

这种策略允许慢速客户端继续完成旧版本下载，同时通过内容寻址复用静态 JAR，限制长期磁盘增长。

## 生命周期与线程模型

- 文件监听线程只消费事件并设置 dirty，不执行哈希或复制。
- 构建、清理和全量校验在 `ThreadPools.io()` 执行。
- 防抖与周期任务由 `ThreadPools.scheduler()` 调度。
- 快照发布使用 volatile 引用；已发布对象不可变，网络线程可以无锁读取。
- 服务停止时取消监听、校验和防抖任务，等待正在执行的构建自然结束但禁止再次发布。

## 错误处理

- 初次快照构建失败：`envReady=false`，客户端按现有退避逻辑重试。
- 后续构建失败：记录错误并保留上一版快照。
- Blob 缺失或哈希不符：将该快照标为不可服务，返回明确错误并立即触发重建。
- 客户端请求不存在的快照：返回 `snapshot_not_found`，客户端重新拉清单。
- 文件持续变化超过三次：本轮构建失败，避免发布可能处于写入中间态的文件。

## 测试策略

新增 `src/test/java` 单元与文件系统集成测试：

- 清单生成后修改 `level.dat`，旧 `snapshotId` 仍下载到旧内容。
- 新快照发布后，旧快照和新快照并发读取互不影响。
- 相同静态文件跨快照复用同一个 Blob。
- 文件复制期间变化会重试，持续变化不会发布半成品。
- 目标端过滤后的清单仍携带完整快照 ID。
- 不存在或已回收的快照返回 `snapshot_not_found`。
- 监听防抖只触发一次构建，60 秒校验能发现漏报变化。
- 中转缓存更新期间继续提供上一份完整快照。

完成后运行 `./gradlew test` 与 `./gradlew build`，并用实际服务端验证 `level.dat` 更新期间主客户端仍能完成环境同步和角色切换。
